# 바바라 민토 피라미드 원칙 (Minto Pyramid Principle)

> 기반: Barbara Minto, *The Minto Pyramid Principle* (1987)

## 핵심 철학

**결론부터 말하고, 근거를 MECE하게 구조화하라**

독자는 바쁘다. 결론을 먼저 제시하고, 그 근거를 논리적으로 그룹화해서 전달해야 한다.

---

## 3가지 핵심 원칙

### 1. Top-Down: 결론 먼저 (Situation → Complication → Question → Answer)

**SCQA 프레임워크**로 도입부를 구성:

| 단계 | 역할 | 예시 |
|------|------|------|
| **S**ituation | 독자가 아는 상황 | "Flutter Web으로 서비스를 운영 중이다" |
| **C**omplication | 문제/변화 | "SEO가 불가능하고 성능이 나쁘다" |
| **Q**uestion | 자연스러운 질문 | "어떻게 해결할 것인가?" |
| **A**nswer | 핵심 결론 | "Next.js로 마이그레이션한다" |

```
# 적용 예시

## 결론 (Answer)
Flutter Web → Next.js 마이그레이션은 기술적으로 필수였고, 5개월 만에 성공적으로 실행했다.

## 근거 1: WHY - 왜 필수였나?
## 근거 2: EVIDENCE - 얼마나 심각했나?
## 근거 3: HOW - 어떻게 실행했나?
## 근거 4: RESULT - 결과는 어떤가?
```

### 2. MECE: 겹치지 않고 빠짐없이 (Mutually Exclusive, Collectively Exhaustive)

모든 하위 그룹은 **서로 겹치지 않으면서(ME)**, **합치면 전체를 커버(CE)**해야 한다.

#### MECE 체크리스트

```
✓ 각 섹션이 독립적인 관점을 다루는가? (ME)
✓ 모든 섹션을 합치면 전체 주장을 완전히 뒷받침하는가? (CE)
✗ 같은 내용이 두 섹션에 반복되지 않는가?
✗ 빠진 관점이 없는가?
```

#### MECE 분류 패턴

| 패턴 | 설명 | 예시 |
|------|------|------|
| **시간순** | Before → During → After | 배경 → 실행 → 결과 |
| **구조순** | 전체를 구성 요소로 분해 | 프론트 / 백엔드 / 인프라 |
| **중요도순** | 가장 중요한 것부터 | 핵심 → 부가 → 참고 |
| **프로세스순** | 단계별 진행 | 분석 → 설계 → 구현 → 검증 |
| **Why-What-How** | 동기 → 내용 → 방법 | 왜 → 무엇을 → 어떻게 |

### 3. 그룹 내 논리 순서 (Logical Order)

같은 레벨의 항목들은 **하나의 논리 순서**를 따라야 한다:

| 순서 유형 | 사용 시점 | 예시 |
|-----------|----------|------|
| **시간순** (Chronological) | 단계/프로세스 설명 | Phase 1 → Phase 2 → Phase 3 |
| **구조순** (Structural) | 부분을 나열할 때 | 프론트 → 백엔드 → 인프라 |
| **중요도순** (Degree) | 비교/평가할 때 | 가장 심각한 문제 → 보통 → 경미 |
| **연역순** (Deductive) | 인과관계 설명 | 전제 → 추론 → 결론 |

## 문서 작성 시 적용 방법

### Step 1: 핵심 메시지 정의

```
Q: 이 문서를 한 문장으로 요약하면?
A: _________________________ (이것이 피라미드 꼭대기)
```

### Step 2: MECE하게 근거 분류 (보통 3~5개)

```
핵심 메시지
├── 근거 1: _________ (왜?)
├── 근거 2: _________ (왜?)
├── 근거 3: _________ (왜?)
└── 근거 4: _________ (왜?)
```

### Step 3: 각 근거에 데이터/사실 배치

```
근거 1
├── 데이터 A
├── 데이터 B
└── 데이터 C
```

### Step 4: Table of Contents로 구조 노출

독자가 **전체 구조를 먼저 파악**할 수 있도록 상단에 목차를 배치.
각 항목에 한 줄 설명을 추가해서 읽지 않아도 흐름을 이해할 수 있게.

---

## 적용 시 주의사항

### DO
- 결론을 **첫 문장**에 배치
- 각 섹션 제목이 **독립적으로 의미 전달** (제목만 읽어도 흐름 파악)
- 동일 레벨 항목은 **같은 문법 구조** (병렬 구조)
- 데이터/수치로 주장 뒷받침

### DON'T
- 배경 설명을 길게 늘어놓고 결론을 마지막에 배치 (Bottom-Up)
- 같은 내용을 다른 섹션에서 반복 (MECE 위반)
- "기타" 항목에 중요한 내용 몰아넣기 (CE 위반)
- 하위 항목 없이 단일 항목만 존재하는 그룹 (최소 2개)

## 코드/기술 문서 적용 예시

### API 문서
```
# Payment API (핵심: 결제 처리를 위한 REST API)

## 1. 인증 (Authentication)
## 2. 결제 생성 (Create Payment)
## 3. 결제 조회 (Get Payment)
## 4. 결제 취소 (Cancel Payment)
## 5. 웹훅 (Webhooks)
```

### 장애 보고서 (Postmortem)
```
# 결론: DB 커넥션 풀 고갈로 30분간 서비스 중단

## 1. 무엇이 발생했나? (Impact)
## 2. 왜 발생했나? (Root Cause)
## 3. 어떻게 대응했나? (Response)
## 4. 재발 방지책은? (Prevention)
```

### 마이그레이션 분석
```
# 결론: Flutter → Next.js 전환은 필수였고, 성공했다

## 1. WHY — 왜 전환이 필수였나?
## 2. EVIDENCE — 얼마나 심각했나?
## 3. HOW — 어떻게 실행했나?
## 4. RESULT — 결과는 어떤가?
```

## 빠른 자가 점검

문서 작성 후 이 질문들로 검증:

1. **피라미드 꼭대기**: 핵심 메시지가 첫 문단에 있는가?
2. **MECE**: 각 섹션이 겹치지 않고, 합치면 전체를 커버하는가?
3. **So What?**: 각 데이터/사실에 "그래서 뭐?" 라고 물었을 때 답이 있는가?
4. **논리 순서**: 같은 레벨 항목들이 일관된 순서를 따르는가?
5. **제목 독립성**: 목차만 읽어도 문서 흐름을 이해할 수 있는가?
