180

→ ×180 감소
1
After
setState 호출 / 드래그 1회
왜 이 가이드인가? 네이버맵 SDK는 명령형(imperative) 라이브러리입니다. React의 선언형 상태 관리와 충돌하면 지도 이벤트 하나가 수십 번의 리렌더링을 유발합니다. 이 가이드는 그 충돌 지점을 시각화하고, 각 최적화 기법을 직접 눈으로 확인합니다.
// 01

문제 진단: 이벤트 폭풍

지도를 드래그하면 bounds_changed, zoom_changed, center_changed 이벤트가 동시에 연속 발화됩니다. 각 이벤트 핸들러에서 Store를 업데이트하면 렌더링 폭풍이 일어납니다.

인터랙티브 데모 — 지도 드래그 시뮬레이션
Before: 3가지 이벤트 구독
bounds_changed
zoom_changed
center_changed
setState 호출 / 초 0
각 이벤트마다 개별 setState → 폭발적 리렌더링
After: idle 이벤트 단독 구독
idle
setState 호출 / 제스처 0
드래그 완료 후 단 1회 — 불필요한 중간 상태 없음
✕ Before — 이벤트마다 구독

✓ After — idle 단독 구독

배치 업데이트: syncMapState

이벤트를 하나로 줄여도, 핸들러 내부에서 여러 번 setState하면 리렌더링이 여러 번 발생합니다. Immer의 단일 produce로 모든 상태를 한 번에 변경합니다.

Before: 분산된 setState
idle 이벤트 → updateMapData()
↓
set(produce()) #1 → 리렌더

idle 이벤트 → updateCameraPosition()
↓
set(produce()) #2 → 리렌더
set(produce()) #3 → 리렌더
After: 단일 syncMapState
idle 이벤트 → syncMapState()
↓
set(produce()) — 단 1회 → 1회 리렌더
Immer produce로 draft에서
모든 필드를 한 번에 변경.
불변성 보장 + 단일 setState.
✕ Before — 분산 setState
// updateMapData set(produce(draft => { draft.map.bounds = bounds; })); // updateCameraPosition (별개 함수) set(produce(draft => { draft.map.center = center; draft.map.zoom = zoom; })); // 총 2번 set → 2번 리렌더링
✓ After — 단일 syncMapState
const syncMapState = ( bounds: naver.maps.Bounds, center: naver.maps.LatLng, zoom: number ) => { set(produce(draft => { // 한 번의 produce로 모두 처리 draft.map.bounds = bounds; draft.map.center = center; draft.map.zoom = zoom; })); }; // 1번 set → 1번 리렌더링

Zustand 셀렉터 최적화

useRootStore()로 스토어 전체를 구독하면, mapInstance 하나가 바뀌어도 좌표 데이터와 무관한 모든 컴포넌트가 리렌더됩니다. 정밀한 셀렉터로 구독 범위를 최소화합니다.

인터랙티브 데모 — 셀렉터에 따른 리렌더링 범위
const store = { lat: 37.5665, lng: 126.9780, x1: 126.90, y1: 37.52, x2: 127.05, y2: 37.62, zoom: 14, mapInstance: "NaverMap#a1b2", projection: Projection, streetData: StreetData[] }
Before: useRootStore() 전체 구독
OverlayLayer좌표 사용

After: useShallow() 정밀 셀렉터
OverlayLayerlat, lng만 구독

✕ Before — 전체 구독
// 스토어 전체를 구독 → 무엇이 바뀌든 리렌더 const { lat, lng } = useRootStore( (state) => state.map // 객체 반환 → 항상 새 참조! ); // mapInstance가 바뀌어도 // lat/lng만 쓰는 컴포넌트가 리렌더됨
✓ After — useShallow 정밀 셀렉터
import { useShallow } from 'zustand/react/shallow'; // 필요한 필드만 얕은 비교로 구독 const { lat, lng } = useRootStore( useShallow((state) => ({ lat: state.map.lat, lng: state.map.lng, })) ); // mapInstance 변경 시 이 컴포넌트는 무반응!

구독 vs 1회 읽기

모든 Store 접근이 구독일 필요는 없습니다. NaverMap 초기화 시 필요한 center/zoom은 마운트 시 단 1회만 읽으면 됩니다. 이후 지도 상태는 SDK가 관리하기 때문입니다.

구독 (Subscribe) — 잘못된 사용
Component
←→
값 바뀔 때마다
리렌더
Store
지도가 이동할 때마다 center가 업데이트되고, 컴포넌트가 매번 리렌더됩니다. 하지만 NaverMap 초기화는 이미 완료되어 이 값이 필요 없습니다.
1회 읽기 (Read Once) — 올바른 사용

(마운트 시)
←
마운트 시

초기화에 필요한 값만 한 번 읽습니다. 이후 상태 변경은 구독하지 않으므로 불필요한 리렌더링이 없습니다.
✕ Before — 불필요한 구독
const NaverMapView = () => { // center가 바뀔 때마다 컴포넌트 리렌더! const { initialCenter, initialZoom } = useRootStore(useShallow(state => ({ initialCenter: state.map.center, initialZoom: state.map.zoom, }))); useEffect(() => { initMap({ center: initialCenter, zoom: initialZoom }); }, [initialCenter, initialZoom]); // 매번 재초기화! };
✓ After — 마운트 시 1회 읽기
const NaverMapView = () => { const mapRef = useRef(null); useEffect(() => { // getState()로 현재 값 1회 읽기 — 구독 X const { center, zoom } = useRootStore.getState().map; mapRef.current = new naver.maps.Map(el, { center, zoom, }); }, []); // 마운트 1회만 실행 };

DOM 객체 재사용

SDK 객체(Polyline, Marker 등)는 생성할 때마다 DOM 노드와 이벤트 바인딩이 발생합니다. 60fps로 경로가 업데이트되면 초당 60개의 객체가 생성·파괴됩니다. 대신 setPath()로 기존 객체를 재사용합니다.

인터랙티브 데모 — Polyline 생성/재사용
현재: Before 모드 (생성/파괴 반복)
생성: 0
파괴: 0
재사용: 0
✕ Before — 매번 새 객체 생성
// 경로 데이터가 바뀔 때마다 호출됨 (60fps!) const updatePolyline = (path: LatLng[]) => { // 기존 객체 파괴 if (polylineRef.current) { polylineRef.current.setMap(null); // DOM 제거 polylineRef.current = null; } // 새 객체 생성 — DOM 노드 + 이벤트 바인딩 polylineRef.current = new naver.maps.Polyline({ path, map, }); };
✓ After — setPath()로 재사용
// 최초 1회만 생성 const initPolyline = () => { polylineRef.current = new naver.maps.Polyline({ path: [], map, }); }; // 이후 경로 변경은 setPath()로만 const updatePolyline = (path: LatLng[]) => { polylineRef.current?.setPath(path); // DOM 재사용 — 생성/파괴 비용 없음 };

renderToStaticMarkup 캐싱

마커 아이콘을 JSX로 렌더링할 때 renderToStaticMarkup()은 매번 Virtual DOM을 생성하고 문자열로 변환합니다. 동일한 입력이면 결과도 동일하므로 캐시가 효과적입니다.

최종 정리: 3가지 원칙

모든 최적화는 세 가지 원칙으로 귀결됩니다. 이 원칙은 NaverMap뿐 아니라 Chart.js, CodeMirror, Three.js 등 모든 명령형 라이브러리와 React 연결에 동일하게 적용됩니다.

01 / PRINCIPLE
최소 빈도
Minimum Frequency
이벤트 발화 빈도를 줄입니다. 연속 이벤트 대신 완료 이벤트(idle)를 구독하여 사용자 제스처 1회당 단 1번의 상태 업데이트가 일어나게 합니다.
idle event · debounce · throttle
02 / PRINCIPLE
최소 범위
Minimum Scope
상태 변경의 파급 범위를 줄입니다. useShallow 셀렉터로 실제 사용하는 필드만 구독하여, 무관한 상태 변경에는 반응하지 않습니다.
useShallow · selector · memoize
03 / PRINCIPLE
최소 구독
Minimum Subscription
구독 자체를 줄입니다. 초기화에만 필요한 값은 getState()로 1회만 읽고, SDK 객체는 재생성 대신 setter로 재사용합니다.
getState · setPath · DOM reuse
이 원칙은 모든 imperative 라이브러리(지도, 차트, 에디터, 3D 뷰어)와 React 연결에 동일하게 적용됩니다.
"React가 관리하는 영역"과 "SDK가 관리하는 영역"을 명확히 분리하는 것이 핵심입니다.
